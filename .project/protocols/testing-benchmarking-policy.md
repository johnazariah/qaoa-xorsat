# Testing, Experimentation, and Benchmarking Policy

## Purpose

Define the canonical policy for:

- correctness verification
- exploratory experimentation
- controlled benchmarking
- published-result reproduction
- persistent result preservation

This document is the durable source of truth for how QAOA-XORSAT testing and
performance measurement are run.

It supersedes ad hoc handoff notes for testing and CI.

For a formal explanatory walkthrough of the testing stack, see
`.project/protocols/testing-protocol.md`.

For the methodological protocol used to structure experiments, benchmarks, and
paper-grade numerical evidence, see
`.project/protocols/experimentation-benchmarking-protocol.md`.

For the distinct protocol used to validate against external targets, see
`.project/protocols/reproduction-protocol.md`.

## Principles

1. Correctness and benchmarking are different activities.
2. Correctness checks should be cheap, frequent, and trusted.
3. Performance claims should come from controlled runs on the dedicated testbed.
4. Historical results must be append-only and provenance-rich.
5. Benchmark data should not pollute normal source-control history.

## Terms

### Verification

Fast checks that answer: does the code still work?

Examples:

- unit tests
- cross-validation tests
- property tests
- published-value regression tests that fit ordinary CI budgets

### Experimentation

Exploratory runs used to learn what might work.

Examples:

- trying new optimiser budgets
- comparing seeds
- probing deeper `p` values
- running local dirty-tree experiments

Experiment results are useful, but they are not benchmark-grade by default.

### Benchmarking

Controlled comparative runs used to detect improvement or regression over time.

Benchmark runs must hold fixed:

- machine class
- runner label
- core parameter range
- optimiser budget
- RNG seed policy
- reporting schema

### Reproduction

Deliberate runs that validate a claim against an external target, such as Farhi
2025 Table 1, before computing project-target values.

## Environment Policy

### Development machine

The development machine is for:

- editing code
- quick local tests
- small exploratory probes
- debugging failed runs

It is not the canonical timing environment because ambient load distorts runtime
comparisons.

### Dedicated testbed

The dedicated 48 GB machine is the canonical environment for:

- benchmark runs
- scheduled experiment sweeps
- reproduction runs
- regression tracking

The reason is measurement quality, not only memory capacity. The testbed reduces
noise from unrelated workload, editor activity, indexing, browsers, and thermal
variance on the development machine.

## Workflow Policy

### CI verification

Ordinary CI is for correctness verification only.

CI should:

- run on every push and pull request to protected branches
- remain fast enough for normal development
- avoid heavy optimisation sweeps
- avoid writing benchmark history

CI success is a prerequisite for benchmark-grade claims, but benchmarking is not
part of ordinary CI.

### Experiment workflows

Experiment workflows are manually triggered or intentionally scheduled. They may:

- run on hosted infrastructure for cheap probes
- run on the dedicated testbed for heavier sweeps
- preserve results if the run is worth keeping

Experiment runs may originate from `git_dirty=true` working trees, but that fact
must be recorded.

### Benchmark workflows

Benchmark workflows are deliberate and controlled.

They must:

- run on the dedicated testbed
- start from committed code
- use a fixed protocol
- preserve results append-only
- be comparable across time

Benchmark workflows should be triggered manually or on a controlled schedule,
not on every code change.

### Reproduction workflows

Reproduction workflows validate against external targets first, then compute the
project-target values. They are deliberate runs, not ordinary CI.

## Result Preservation Policy

### Canonical storage layout

Persistent optimisation and benchmark artefacts live under:

`/.project/results/optimization/`

with two layers:

1. `index.csv`
2. `runs/<run_id>/`

The canonical unit of preservation is the per-run directory, not the aggregate
CSV.

Each deliberate run should create its own immutable directory under
`runs/<run_id>/`. That directory is the authoritative provenance record for the
run.

The aggregate `index.csv` exists as a derived convenience layer for cross-run
analysis, filtering, and charting. It is not the primary source of truth.

### Per-run directory contents

Each run directory should contain, at minimum:

1. `manifest.json`
2. `results.csv`

`manifest.json` is for structured metadata and provenance.

`results.csv` is for the tabular per-depth outcome data generated by that run.

This hybrid structure is preferred over either extreme:

- not a single giant CSV as the only archive
- not isolated run files with no aggregate index

The per-run directory is the canonical record. The aggregate index is the
analysis surface.

### Append-only rule

Historical results are append-only.

- Never overwrite an old run directory.
- Never rewrite old rows in `index.csv`.
- If a run is later judged invalid, classify it downstream rather than mutating
  the raw archive.

If the aggregate index is ever rebuilt, it must be rebuilt from the canonical
per-run directories rather than treated as independently authoritative data.

### Run classes

Every preserved run must be classifiable as one of:

- `experiment`
- `benchmark`
- `reproduce`

This distinction is mandatory for honest analysis. Exploratory runs must not be
mixed blindly into benchmark regression charts.

### Required provenance fields

Every preserved run must record at minimum:

- `run_id`
- `run_kind`
- `timestamp_utc`
- `git_commit`
- `git_branch`
- `git_dirty`
- runner or machine label
- `k`, `D`, and `p`
- `clause_sign`
- restart budget
- iteration budget
- seed
- value
- total wall-clock time
- best-start wall-clock time when multistart is used
- evaluation count
- start count
- iteration count
- convergence flag

Without these fields, improvement or regression claims are not defensible.

Where practical, provenance should live first in the per-run `manifest.json`,
with the aggregate index carrying the subset needed for plotting and filtering.

### Interpretation rules

For fixed `(k, D, p)` and fixed benchmark protocol:

- higher `value` is better optimisation quality
- lower `wall_time_seconds` is better runtime performance
- lower evaluation count at equal quality is better optimiser efficiency

`git_dirty=true` runs are informative, but they are not benchmark-grade evidence
for commit-to-commit comparisons.

## Branch Policy For Generated Data

Generated benchmark history should live on a dedicated `results` branch, not on
`main`.

Reasons:

- avoids polluting source history
- avoids retriggering ordinary CI on result-only commits
- reduces push conflicts with source development
- keeps append-only generated data clearly separated from code review history

Source branches such as `main` and feature branches own code. The `results`
branch owns machine-generated benchmark history.

## Runner Policy

The dedicated testbed runner should be repository-scoped and trusted-only.

It should:

- use labels `self-hosted`, `linux`, `x64`, and `testbed-48gb`
- run as a service
- have Julia installed on `PATH`
- be protected from untrusted PR execution
- run one benchmark-grade job at a time

Benchmark and reproduction workflows should use workflow-level concurrency so
testbed jobs do not overlap.

### Specific 48 GB runner setup

The preferred dedicated runner host is:

- Ubuntu 24.04 LTS Server
- no desktop environment
- dedicated repository-scoped runner user
- Julia installed from the official binaries and available on `PATH`

Recommended base packages:

```bash
sudo apt-get update
sudo apt-get install -y git curl wget tar xz-utils ca-certificates
```

Create a dedicated runner account:

```bash
sudo useradd --create-home --shell /bin/bash gha-runner
sudo passwd -l gha-runner
```

Install Julia under `/opt` and expose it on `PATH`:

```bash
cd /tmp
wget https://julialang-s3.julialang.org/bin/linux/x64/1.11/julia-1.11.7-linux-x86_64.tar.gz
sudo tar -C /opt -xzf julia-1.11.7-linux-x86_64.tar.gz
sudo ln -sfn /opt/julia-1.11.7 /opt/julia
echo 'export PATH=/opt/julia/bin:$PATH' | sudo tee /etc/profile.d/julia.sh
```

Install the GitHub Actions runner as the dedicated user and register it with:

- repository scope
- labels `self-hosted,linux,x64,testbed-48gb`
- service mode

The runner should live under `/home/gha-runner/actions-runner/` and should be
installed with the GitHub-provided `svc.sh` service wrapper.

Operational hardening on the laptop host:

- keep the machine on AC power during benchmark-grade runs
- disable suspend on lid close and automatic sleep while on AC
- keep a small swap or `zram` safety buffer rather than disabling swap entirely
- avoid unrelated user sessions during benchmark runs
- verify thermals and CPU governor behaviour before treating results as benchmark-grade

### Reliability capture for benchmark-grade runs

Benchmark and reproduction runs on the dedicated testbed should preserve machine
state snapshots before and after the heavy command.

This repository provides:

- `scripts/testbed/capture-machine-state.sh`
- `scripts/testbed/run-with-machine-state.sh`
- `scripts/testbed/analyze-machine-state.sh`

The wrapper script should be preferred for benchmark-grade runs because it:

- captures a pre-run machine snapshot
- runs the requested command while recording `/usr/bin/time -v` output when available
- captures a post-run machine snapshot
- writes a small run summary with timestamps and exit status

The minimum reliability artefacts for a benchmark-grade run are:

- `machine-state-before.json`
- `machine-state-after.json`
- `command.txt`
- `command-time.txt` when available
- `stdout.txt`
- `stderr.txt`
- `run-summary.json`
- `machine-analysis.json` after analysis

These artefacts are used to judge whether a run was likely affected by thermal
pressure, unexpected background load, low-memory conditions, or power-state changes.

The analyzer script should be run after the wrapper completes so the run record
contains an explicit machine-health assessment alongside the raw snapshots.

### Machine-state signals to inspect

At minimum, inspect:

- load averages
- available memory and swap
- top CPU-consuming processes
- CPU governor and average current frequency
- thermal-zone temperatures
- AC and battery state on laptop hardware

Runs with obviously abnormal pre-run or post-run machine state should be marked
downstream rather than deleted from the raw archive.

## Acceptance Criteria

This policy is in force when all of the following are true:

1. Ordinary CI runs correctness verification only.
2. Benchmarking and reproduction are deliberate workflows, not push-triggered CI.
3. The dedicated 48 GB testbed is the canonical timing environment.
4. Preserved results use the append-only optimisation archive schema.
5. Historical benchmark data is stored on a dedicated `results` branch.
6. Regression comparisons are made only between comparable benchmark-grade runs.
