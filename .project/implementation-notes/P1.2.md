## 1. Scope & Objective

P1.2 translates the QAOA bra-ket sandwich ⟨γ,β|C_α|γ,β⟩ into concrete tensor
primitives suitable for contraction on the light-cone tree defined by P1.1.

## 2. Design Decisions

### 2.1 Hyperindex Convention: Interleaved Ordering

The spec left the hyperindex bit ordering open. We chose **interleaved**:

```
(ket₁, bra₁, ket₂, bra₂, …, ket_p, bra_p)
```

rather than the grouped alternative `(ket₁, …, ket_p, bra₁, …, bra_p)`.

**Rationale**: Interleaving keeps each QAOA round local to a contiguous bit-pair
`(2ℓ-1, 2ℓ)`. This is critical for P1.3's contraction loop, which processes one
round at a time — with interleaving, extracting or masking a round's bits is a
single 2-bit window operation, whereas the grouped layout would require
straddling two distant bit regions.

The mapping is:
- Round `ℓ` → bit positions `(2ℓ-1, 2ℓ)`
- Root (innermost) slice → round 1 → bits `(1, 2)`
- Leaf (outermost) slice → round `p` → bits `(2p-1, 2p)`
- Total hyperindex range: `0` to `4^p - 1` (a `2p`-bit integer)

### 2.2 Raw Tensors, Not Effective Tensors

The implementation provides **raw local tensors** matching Farhi 2025 Eq. (13).
These are the building blocks that P1.3 will compose into effective branch-transfer
objects during contraction. Crucially:

- Leaf and observable tensors are **real-valued** (`Vector{Float64}`)
- Mixer and problem tensors are **complex-valued** (`Matrix{ComplexF64}`,
  `Vector{ComplexF64}`)

The spec (acceptance criterion 4) suggested all tensors should be real, following
the paper's claim that the bra-ket combination eliminates complex phases. This is
true for the *final contracted expectation value*, but not for individual raw
tensors — the mixer superoperator inherently carries `±i sin β` entries, and the
problem phase is `cis(γ(z_bra - z_ket)/2)`. These intermediate complex values
cancel only after full contraction to a scalar.

**Decision**: Keep mixer and problem tensors complex. Document this as a
deliberate spec deviation. The end-to-end P1.3 validation (MaxCut p=1 →
c̃_edge ≈ 0.7500) will confirm correctness.

### 2.3 Diagonal Storage for Problem & Observable Tensors

Both the problem tensor and the observable tensor are diagonal in the
computational basis — they assign weights to hyperindex *configurations* but
don't couple different configurations. Storing only the diagonal as a flat
`Vector` saves a factor of `(4^p)^k` in memory versus a full `(4^p)^k × (4^p)^k`
matrix:

```
problem_tensor(k, γ, round, p)  → Vector{ComplexF64} of length (4^p)^k
observable_tensor(k, p)         → Vector{Float64}     of length (4^p)^k
```

To recover the multi-index structure, reshape:
```julia
reshape(problem_tensor(k, γ, round, p), ntuple(_ -> 4^p, k)...)
```

### 2.4 Mixer as Full Matrix

Unlike the diagonal tensors, the mixer superoperator *does* couple different
hyperindex values (it flips bits within a specific round). It is stored as a
dense `4^p × 4^p` matrix:

```
mixer_tensor(β, round, p) → Matrix{ComplexF64} of size (4^p, 4^p)
```

**Scalability note**: This materialises `16^p` entries. At p=5, that's
~1M entries — acceptable. At p=10+, P1.3 should compute the mixer's effect
on-the-fly via bit masking rather than materialising the full matrix. This is
flagged as a P1.3/P1.4 optimisation concern, not a P1.2 issue.

## 3. Mathematical Derivations

Full derivations live in `.project/learning/05-tensor-derivation.md`. This
section summarises the key results and their verification.

### 3.1 Leaf Tensor

**Claim**: `L(σ) = 2^{-p}` for all hyperindex values σ, independent of angles.

**Derivation**: The boundary qubit starts in |+⟩. At each QAOA round ℓ, the
bra-ket sandwich contribution for a single qubit is:

```
⟨+|e^{iβ_ℓ X}|b_ℓ⟩ · ⟨k_ℓ|e^{-iβ_ℓ X}|+⟩
```

Since `X|+⟩ = |+⟩`, we have `e^{-iβX}|+⟩ = e^{-iβ}|+⟩`, so:

```
⟨k_ℓ|e^{-iβ_ℓ X}|+⟩ = e^{-iβ_ℓ} ⟨k_ℓ|+⟩ = e^{-iβ_ℓ}/√2
```

Similarly `⟨+|e^{iβ_ℓ X}|b_ℓ⟩ = e^{iβ_ℓ}/√2`. The product is `1/2`
regardless of `k_ℓ`, `b_ℓ`, or `β_ℓ`. Over `p` rounds: `(1/2)^p = 2^{-p}`.

**Verification**: Tested for p=1..4 with random angles — every entry equals
`2^{-p}` to machine precision. ✓

**Caveat**: The exact normalisation constant depends on the contraction
convention P1.3 adopts. If P1.3 uses a different initial-state normalisation
(e.g., absorbing `1/√2` factors into the mixer), this value may need adjustment.
Must be validated end-to-end.

### 3.2 Mixer Tensor

**Formula**: For round ℓ acting on bit positions `(2ℓ-1, 2ℓ)`:

```
M(σ_out, σ_in) = U_X(β)_{k_out, k_in} · conj(U_X(β)_{b_out, b_in})
```

where `U_X(β) = e^{-iβX} = [cos β, -i sin β; -i sin β, cos β]`, and `k_out`,
`k_in` are the ket bits of round ℓ, `b_out`, `b_in` the bra bits. All other
round-bits are unchanged (identity).

At `p = 1`, `β = π/6`, the 4×4 matrix first column is:

| σ | (k_out, b_out) | Entry | Value |
|---|-----------------|-------|-------|
| 0 | (0,0) | cos²β | 3/4 |
| 1 | (1,0) | -i·cos·sin | -i√3/4 |
| 2 | (0,1) | +i·cos·sin | +i√3/4 |
| 3 | (1,1) | sin²β | 1/4 |

**Verification**:
- Golden values at β=π/6 match hand computation ✓
- `M·M† = I` (superoperator unitarity) verified for p=1,2,3 ✓
- `M × [1,1,1,1] = [1,1,1,1]` (trace preservation) ✓
- M(β=0) = I (identity at zero angle) ✓
- Multi-round: M at round 2, p=2 correctly acts on bits 3,4 only ✓

### 3.3 Problem Tensor

**Formula**: For k qubits at round ℓ with angle γ:

```
P(σ₁, …, σ_k) = cis(γ · (z_bra - z_ket) / 2)
```

where:
- `z_ket = ∏ⱼ (-1)^{ket_bit(σⱼ, ℓ)}` (product of Z eigenvalues on ket side)
- `z_bra = ∏ⱼ (-1)^{bra_bit(σⱼ, ℓ)}` (product of Z eigenvalues on bra side)

This combines the ket phase `exp(-iγ Z₁⋯Zₖ/2)` with the conjugate bra phase
`exp(+iγ Z₁⋯Zₖ/2)`.

At p=1, k=2, with all child hyperindices = 0 (i.e. ket=0, bra=0 for both
qubits), the parent-indexed diagonal is:

| σ_parent | (ket, bra) | z_ket × z_child | z_bra × z_child | P |
|----------|------------|-----------------|-----------------|---|
| 0 | (0,0) | +1 | +1 | 1 |
| 1 | (1,0) | -1 | +1 | cis(γ) |
| 2 | (0,1) | +1 | -1 | cis(-γ) |
| 3 | (1,1) | -1 | -1 | 1 |

**Verification**:
- k=2 golden values at γ=π/3 match hand derivation ✓
- k=3 golden values at γ=π/3 match hand derivation ✓
- P(γ=0) = all ones (no phase at zero problem angle) ✓
- 2π-periodic ✓

### 3.4 Observable Tensor

**Formula**: At the root constraint with observable `C_α = (1 + Z₁⋯Zₖ)/2`:

1. If any qubit has `ket₁ ≠ bra₁` (off-diagonal in computational basis) → weight = 0
2. Otherwise: weight = `(1 + ∏ⱼ(-1)^{ket₁(σⱼ)}) / 2`

This yields values in `{0, 1}` — it's a projector onto the satisfying subspace.

**Key detail for k=3 (odd k)**: When all qubits have Z=-1, the product is
`(-1)³ = -1`, so the weight is `(1-1)/2 = 0`. This is different from k=2 (even)
where all-negative gives `(-1)² = +1` and weight = 1. This was initially
mis-tested and caught during review.

**Verification**:
- k=2 p=1: exactly 2 nonzero entries (both = 1) out of 16 ✓
- k=3 p=1: correct parity structure with odd-k sign ✓
- Off-diagonal entries (ket≠bra) always zero ✓

## 4. Implementation Details

### 4.1 Type Hierarchy

```
QAOAAngles (struct)
  ├─ γ::Vector{Float64}   # problem angles, length p
  ├─ β::Vector{Float64}   # mixer angles, length p
  └─ inner constructor validates length match and p ≥ 1

depth(angles) → Int       # convenience accessor
```

The constructor accepts `AbstractVector{<:Real}` and promotes to `Float64`,
handling Julia's type promotion gracefully (e.g., `QAOAAngles([1, 2], [3, 4])`
works with integer inputs).

### 4.2 Validation Helpers

Three small validators are used pervasively:

```julia
validate_depth(p)              # p ≥ 1
validate_round(round, p)       # 1 ≤ round ≤ p
validate_hyperindex(σ)         # σ ≥ 0
```

These throw `ArgumentError` with descriptive messages. The `validate_hyperindex`
function accepts `Integer` (not just `Int`) to handle Julia binary literals
which are `UInt8` (see §5.1).

### 4.3 Hyperindex Bit Manipulation

```julia
hyperindex_bit(σ, ℓ)           # extract 1-indexed bit ℓ from σ
hyperindex_parity(σ, positions) # XOR parity of selected bits
round_bit_positions(round, p)  # → (2round-1, 2round)
```

`hyperindex_bit` accepts `Integer` for σ and uses `Int(σ)` internally for bit
shifting. The 1-indexed convention aligns with Julia's general preference.

### 4.4 Internal Helpers

| Function | Purpose |
|----------|---------|
| `z_eigenvalue(bit)` | Maps `{0,1}` → `{+1,-1}` (Z basis eigenvalue) |
| `mixer_gate_entry(out, in, β)` | Single element of `U_X(β)` |
| `ket_bit(σ, round, p)` | Shorthand for hyperindex_bit at ket position |
| `bra_bit(σ, round, p)` | Shorthand for hyperindex_bit at bra position |
| `parity_sign(config, pos)` | Product of Z eigenvalues across k qubits |
| `problem_phase(config, γ, round, p)` | `cis(γ(z_bra - z_ket)/2)` for one entry |
| `observable_weight(config, p)` | Diagonal weight of `(1+Z₁⋯Zₖ)/2` |

These are module-internal (not exported). They follow the Julia convention of
small composable functions — each does exactly one thing.

### 4.5 Mixer Implementation Strategy

The mixer tensor is built via explicit bit manipulation rather than Kronecker
products:

1. For each input hyperindex, extract the round's ket/bra bits
2. Compute the `keep_mask` (all bits except the round's pair)
3. Enumerate all 4 output combinations for the round's bits
4. Place `U_X(β)_ket · conj(U_X(β))_bra` at the appropriate matrix position

This approach has O(4^p × 4) = O(4^{p+1}) complexity, which is optimal for
building the full matrix. The bitmask trick (line 122):

```julia
keep_mask = xor(typemax(Int), ket_mask | bra_mask)
```

efficiently preserves all bits except the target round's pair.

## 5. Julia-Specific Lessons

### 5.1 Binary Literals are UInt8

Julia's binary literals (`0b1010`) produce `UInt8`, not `Int`. This caused
initial `MethodError` failures when `hyperindex_bit` and `hyperindex_parity`
were typed as `Int`. **Fix**: Widen signatures to `Integer`, the abstract
supertype of all integer types.

### 5.2 isapprox Inside Lambdas

The expression `all(x -> x ≈ val atol=tol, vec)` is a parse error — Julia
interprets `atol=tol` as a keyword argument to `all`, not to `≈`. **Fix**: Use
the functional form `isapprox(x, val; atol=tol)` inside lambdas.

### 5.3 LinearAlgebra in Test Extras

Julia 1.12 requires stdlib packages used in tests to be listed in
`Project.toml`'s `[extras]` and `[targets]` sections. We added `LinearAlgebra`
there since `test/test_tensors.jl` uses `using LinearAlgebra` for the identity
matrix `I` in unitarity checks.

### Deviation from Spec: Complex-Valued Tensors

Acceptance criterion 4 states "All tensors are real-valued". Our mixer and
problem tensors are `ComplexF64`. This is mathematically necessary for the raw
local tensors — the complex phases only cancel after full contraction. The spec's
criterion appears to describe the *effective* branch-transfer tensors that
emerge after contraction, not the raw building blocks. This distinction is
documented in `05-tensor-derivation.md`.

## 10. Forward-Looking Notes for P1.3

### 10.1 Contraction Ordering

Farhi 2025 Eq. (14) and Fig. 5(b) specify: **contract box, then exponentiate**.
The P1.3 spec draft (lines 224–264) flags an ambiguity about whether to
exponentiate *then* apply the gate, or contract *then* exponentiate. The correct
order per the paper is:

1. Start from leaf tensor at round `p`
2. At each level moving inward (p → 1):
   a. Contract the appropriate tensor (mixer or problem) with the branch vector
   b. Raise the result element-wise to the branching multiplicity (`D-1` at variable nodes, `k-1` at constraint nodes)
3. Apply the observable at round 1

This ordering is critical for correctness and must be validated by the MaxCut
p=1 golden result (c̃_edge ≈ 0.7500).

### 10.2 Scalability Path

Current memory cost per tensor:
- Leaf: O(4^p) — manageable
- Mixer: O(16^p) — expensive beyond p≈7
- Problem: O((4^p)^k) — very expensive for large p and k

For P1.3/P1.4, the mixer should be applied on-the-fly (multiply-by-column
rather than materialise full matrix). The problem tensor's diagonal structure
means it can also be computed entry-by-entry during contraction. These
optimisations don't change the API but would require P1.3 to use a different
calling pattern than simple matrix-vector multiply.

### 10.3 Leaf Normalisation

The leaf tensor value `2^{-p}` is correct given our derivation, but the P1.3
contraction must use a consistent normalisation convention. If the contraction
loop internally normalises or re-scales, the leaf constant may need adjustment.
This **must** be validated by the end-to-end MaxCut test before P1.2 can be
considered fully proven.
