## 1. Scope & Objective

The original P1.3 draft aimed directly at the `O(4^p)` branch-transfer
contraction from Farhi et al. 2025. During implementation review, one part of
that draft turned out to be mathematically wrong for k-XORSAT:

- variable branching does exponentiate entrywise
- constraint branching does **not** exponentiate entrywise in general

Because of that, this branch first implemented the strongest correctness-first
deliverable that could be defended from the derivation state:

1. an **exact light-cone statevector evaluator** for small trees
2. regression tests against known MaxCut formulas and exact references
3. explicit documentation of the remaining blocker for the future transfer
   recursion

That remains the foundation, but the situation has changed in two important
ways:

1. this branch now also contains an exact finite-D branch-tensor kernel in
   `src/basso_finite_d.jl`, with tests for the Basso Eq. (8.7) update itself
2. the new WHT discovery in `.project/learning/15-wht-factorisation-discovery.md`
   shows that the k-body constraint fold factors as a convolution on
   `(ℤ₂^{2p+1}, ⊕)`, and that factorisation is now wired into the production
   finite-D branch-step path in `src/basso_finite_d.jl`

The main unresolved issue on this branch is therefore narrower than before: the
exact finite-D **root observable** still needs a clean translation into repo
conventions, but the branch recursion, the WHT-backed exact fold, and the
root-side decomposition are now materially clearer.

## 3. Design Decisions

### 3.1 Exact light-cone simulation before transfer compression

The branch now constructs the full finite light cone for `(k, D, p)` and applies
the alternating QAOA problem and mixer layers directly to the `2^n` statevector.
This is exact and easy to validate.

That choice is intentionally conservative. The intended `O(4^p)` recursion is
still the long-term target, but implementing it before the transfer object is
derived would produce a fast wrong answer instead of a slow correct one.

### 3.1a Exact branch kernel before full finite-D evaluator

This branch now also isolates the exact finite-D branch update from the full
finite-D evaluator. That split has been useful in practice.

The current helper layer in `src/basso_finite_d.jl` validates the Basso-style
branch recursion itself without claiming that the final root observable is fully
settled. In particular, it now uses the signed Basso ordering

`(a^[1], …, a^[p], a^[0], a^[-p], …, a^[-1])`

instead of the earlier shortcut that incorrectly treated the first `2p` bits as
the support of `Γ`. The central `a^[0]` slot is part of the root observable,
not the phase dot product.

### 3.2 Experimental MaxCut transfer port as a side track

The branch now also contains an **internal** Julia port of the public
`large-girth-maxcut-qaoa` compact MaxCut recursion. This is deliberately kept
separate from the exported finite-tree evaluator.

That port is useful for three reasons:

1. it captures a working compact recursion in native Julia,
2. it gives the repository a concrete target for future optimisation work on the
   MaxCut side,
3. it makes it easier to compare the exact finite-tree route with the upstream
   large-girth recursion without conflating them.

It is **not** yet used to implement `qaoa_expectation`, because its scalar
objective is not the same object as the branch's exact finite-`D` root-clause
satisfaction value.

### 3.2a WHT factorisation changes the strategic target

The new WHT result matters more than a normal optimisation note. It changes the
expected shape of the exact finite-D algorithm.

The old reading of Tier 2 was:

- exact finite-D Basso is mathematically clean,
- but the k-body constraint fold costs `O(p · 4^{kp})`,
- so exact `(k=3, D=4)` likely tops out around `p ≈ 5–7`.

The new WHT note shows that the constraint fold is a convolution on
`(ℤ₂^{2p+1}, ⊕)`. That means the exact fold should be computable by
Walsh-Hadamard transforms with cost `O(p · 4^p)` per step and therefore
`O(p² · 4^p)` overall, independent of `k` except for a pointwise power in the
transform domain.

That production fold is now in place in the branch-step implementation. The
remaining work is not to replace the naive exact fold any more; it is to finish
the root observable/evaluator on top of the exact WHT-backed branch tensor.

### 3.2b Root-side decomposition before public finite-D evaluation

Recent work on this branch refactored the root contraction into explicit pieces
instead of one opaque kernel:

- `basso_root_message` now represents the root branch message alone,
- `basso_root_parity_kernel` isolates the root observable parity factor,
- `basso_root_problem_kernel` isolates the clause-sign-aware root phase factor,
- `basso_root_kernel` is the product of those two kernels,
- `basso_root_parity_sum` composes the full WHT-backed root contraction.

That split matters because the branch-fold path is already exact and tested. The
remaining mismatch with the Tier 1 oracle is now localised to the root phase /
observable convention map instead of being buried inside a larger contraction.

### 3.3 Public API remains small

The user-facing functions are:

- `parity_expectation(params, angles; clause_sign)`
- `qaoa_expectation(params, angles; clause_sign)`

Internally, the branch also defines:

- `build_light_cone_tree`
- `simulate_light_cone_state`
- problem/mixer application helpers for the exact statevector

These internal helpers are left unexported because they are scaffolding for the
eventual optimised evaluator, not stable API.

### 3.4 Hard guard on exact tree size

The exact evaluator is guarded by `MAX_EXACT_LIGHTCONE_QUBITS = 22`.

This is not an arbitrary product decision. It prevents the package from silently
attempting a calculation whose statevector would explode in memory. For example,
`TreeParams(3, 4, 2)` already requires 129 qubits in the explicit light cone.

The guard fails loudly with a message that points to the missing transfer
contraction as the reason larger trees are not yet supported.

## 4. Implemented Algorithm

### 4.1 Light-cone construction

`build_light_cone_tree(params)` assigns explicit qubit indices to the finite
factor-tree neighbourhood of the root clause and verifies:

- `tree.qubit_count == total_variables(params)`
- `length(tree.clauses) == total_constraints(params)`

### 4.2 State preparation and evolution

The simulator starts from the uniform `|+⟩^{\otimes n}` state and then applies
the physical QAOA rounds in forward order:

1. problem layer over **all** clauses in the light cone
2. mixer layer over **all** qubits in the light cone

This ordering matches the adopted round convention from P1.2:

- physical round `1` is outermost / closest to the initial state
- physical round `p` is innermost / closest to the observable

### 4.3 Root expectation

The final expectation is measured only on the root clause:

- `parity_expectation` returns `⟨Z₁⋯Z_k⟩`
- `qaoa_expectation` returns `(1 + clause_sign * ⟨Z₁⋯Z_k⟩) / 2`

`clause_sign = -1` reproduces odd clauses such as MaxCut edges.

## 5. Validation

The branch validates the reference evaluator with four kinds of tests.

### 5.1 Zero-angle baseline

For `(k=3, D=4, p=1)` and `(γ, β) = 0`, the evaluator returns `0.5`, matching
the random-assignment baseline.

### 5.2 MaxCut p=1 closed form

For `(k=2, D=3, p=1)` the branch matches the exact formula

`⟨Z_u Z_v⟩ = -sin(4β) cos²(γ) sin(γ)`

and therefore the edge satisfaction

`c_edge = (1 - ⟨Z_u Z_v⟩) / 2`.

### 5.3 MaxCut p=1 optimum

At

- `γ* = atan(1 / sqrt(2))`
- `β* = π / 8`

the branch reproduces

`c_edge = 1/2 + sqrt(3)/9 ≈ 0.69245`.

### 5.4 MaxCut p=2 exact reference

For `(k=2, D=3, p=2)`, the branch compares the exported evaluator against an
independent exact statevector reference built directly inside the tests.

### 5.5 Experimental transfer-port regression

The branch also validates the internal `src/maxcut_transfer.jl` port against the
upstream compact MaxCut implementation by checking:

- exact `(2p + 1) × (2p + 1)` matrix entries at `p=1` and `p=2`,
- the associated scalar transfer objective,
- the broadcast-corner symmetry relations built into the upstream algorithm.

These tests do **not** claim equivalence to the finite-tree evaluator. They only
lock the Julia port to the current MaxCut transfer reference.

### 5.6 Relation to the exact finite-`D` evaluator

The branch now has enough numerical evidence to state something more precise:
the scalar returned by `maxcut_transfer_objective` is **not** the exact finite-`D`
edge satisfaction value.

For `D=3`:

- at `p=1`, `(γ, β) = (0.31, 0.17)`:
  - exact `qaoa_expectation = 0.5869839712`
  - transfer objective `= 0.1804390243`
- at the `p=1` optimum:
  - exact `qaoa_expectation = 0.6924500897`
  - transfer objective `= 0.2315645509`
- at `p=2`, `(γ, β) = ([0.21, 0.64], [0.17, 0.39])`:
  - exact `qaoa_expectation = 0.6885673165`
  - transfer objective `= 0.2262887034`

So the current MaxCut transfer port should be interpreted exactly as intended in
its source lineage: an implementation of the upstream compact large-girth scalar,
not a replacement for the branch's exact finite-tree observable.

### 5.7 Raw multilinear constraint update oracle

The branch now also contains a small exact helper,
`contract_constraint_message(child_messages, γ, slice, p; clause_sign=1)`, which
contracts one raw `k`-body problem tensor against `k - 1` child messages and
returns the parent-facing message.

This helper exists for derivation and validation work, not for performance. Its
purpose is to make the non-root constraint update explicit in code:

- for `k = 2`, the update reduces to an ordinary matrix-vector product,
- for `k > 2`, the update is multilinear in the child messages,
- the update is therefore not equivalent in general to entrywise `.^ (k - 1)`.

That gives later transfer-recursion work a concrete oracle for the smallest
exact finite-`D` message object before attempting any compressed `O(4^p)` form.

### 5.8 Smallest finite-`D` exact target

The branch now also validates the smallest nontrivial finite-`D` XORSAT target,
`(k=3, D=2, p=1)`, against an independent explicit statevector reference built
directly in the tests.

This matters because it is the first exact case where:

- the root clause has arity `k > 2`,
- non-root constraints are present,
- each root variable has exactly one child clause, so the geometry is still
   small enough to inspect by hand.

That test does not yet claim a closed raw-tensor `O(4^p)` recursion. It fixes a
clean outer correctness target while the transfer initialisation at the
boundary/root interface is still being derived.

## 6. What Is Still Open

The open problem is no longer a single monolithic blocker. It is now split into
two separate tasks.

### 6.1 Root observable translation

The first unresolved task is the exact finite-D **root observable**.

The branch recursion itself is now implemented and tested at the one-step level,
but a first attempt to wire the Basso Eq. (8.8) / Eq. (8.16) style root
contraction into repo conventions did not reproduce the exact oracle and was
backed out. That means the remaining ambiguity lives at the root observable /
convention boundary, not in the raw branch fold.

The new bridge helper `basso_configuration_to_hyperindex` narrows this further.
It maps a `(2p + 1)`-bit Basso branch configuration into the repo's raw P1.2
hyperindex basis, and the associated tests now verify the following exact local
statement:

- the root local factor obtained from the raw tensor semantics equals the
   branch-basis kernel `basso_root_kernel(...; clause_sign)` when evaluated at
   `branch_degree = 1`.

That means the local root phase/parity object is now understood in branch space.
The remaining mismatch with the Tier 1 oracle is therefore not a mystery about
how the raw root clause itself acts. It must live in the degree-normalised
finite-D iteration / convention stack around that local object.

An immediate follow-up probe on the smallest exact case `(k=3, D=2, p=1)` also
showed that a naive collapse of the current branch-basis message back to the raw
child-clause hyperindex message does **not** match the exact transfer oracle.
So the remaining gap is not confined to the root local factor. It also includes
the relation between the finite-D branch object and the raw parent-facing
message in hyperindex space.

That same smallest case now has a useful exact branch-basis regression formula:
for `p = 1`, `k = 3`, and branch degree `1`, the finite-D branch tensor is

- `H(a) = 1` when the outer ket/bra bits agree,
- `H(a) = cos(2γ)` when the outer ket/bra bits differ,

and is independent of the central `a^[0]` bit as expected from the zero entry in
`Γ`. This does not by itself produce the raw hyperindex-space transfer oracle,
but it gives a precise local target that any future interface map must respect.

Reading the finite-D proof more carefully sharpens this again: `H_D^(m)(a)` is
not presented there as a raw parent-facing message at all. It is the effective
weight of a depth-`m` contracted descendant subtree as a function of the full
`(2p + 1)`-bit node configuration in the proof's inserted-complete-set basis.
Equivalently, it is a partially contracted subtree partition function in the
expanded branch basis, with the local variable-line sandwich factors carried by
`f(a)` still belonging to that basis story.

That explains why a naive sum over the center bit of `H_D^(m)(a)` or
`f(a)H_D^(m)(a)` is not expected to reproduce the raw hyperindex-space transfer
oracle. A further local clause/variable interface map is still missing.

### 6.2 Root phase mapping to raw tensor semantics

The second unresolved task is narrower than before: express the root phase
factor in a way that matches the repo's raw tensor semantics and the exact Tier
1 oracle.

The branch now has enough structure to test this directly:

- the branch tensor is exact and WHT-backed,
- the root parity factor is isolated,
- the root clause phase factor is isolated,
- the smallest exact overlap cases are already available in `test/test_qaoa.jl`.

So the next derivation step is no longer a broad search over full recursions. It
is a direct comparison between the current `basso_root_problem_kernel` and the
root-clause object induced by the raw tensors / exact oracle conventions.

### 6.3 Next implementation step

The next production step should not be "finish `basso_expectation` exactly as
currently sketched and then optimise later". The WHT result changes the order of
operations.

The recommended implementation sequence is:

1. **Checkpoint the current root-side decomposition**
   - keep the implementation note aligned with the code
   - preserve the exact WHT-backed branch path and the root decomposition tests

2. **Compare the root problem kernel against raw tensor semantics**
   - derive the root-clause phase directly from the P1.2 tensor convention
   - match it to the smallest Tier 1 overlap cases
   - avoid another unconstrained sign/normalisation search

3. **Expose a public exact finite-D evaluator only after the overlap cases pass**
   - add direct Tier 2 vs Tier 1 overlap tests
   - then introduce a public `basso_parity_expectation` / `basso_expectation`
     entry point

This order keeps the already-validated branch/WHT machinery separate from the
still-open root-convention map. That separation matters: the branch is green,
the expensive exact fold is already replaced, and the only remaining high-risk
piece is the root observable.

Two facts are now established:

1. At a variable node, the `D-1` identical child branches contribute an
   entrywise power.
2. At a constraint node, the `k-1` child-variable contributions combine
   **multilinearly**, not via entrywise `.^ (k-1)` in general.
3. The multilinear constraint fold admits a Walsh-Hadamard factorisation, and
   that exact fold is now used in the production finite-D branch step.

This means the original draft P1.3 spec was correct in spirit but wrong in one
of its core update rules.

## 7. Current Status

At the current worktree state, the branch has:

- an exact Tier 1 light-cone oracle,
- an exact finite-D branch tensor with WHT-backed fold,
- explicit internal root helpers,
- tests proving naive-vs-WHT agreement for both branch and root convolutions,
- tests proving the root-kernel decomposition behaves coherently at zero angles
   and under clause-sign conjugation.

What it does **not** yet have is a public exact finite-D expectation evaluator
that matches the Tier 1 oracle on overlap cases. That missing equality is the
next technical target.

## 8. Impact on Later Work

This branch gives P1.4 and later optimisation work a trusted reference oracle
for all small-tree cases. Any future `O(4^p)` contraction must first agree with
this implementation on:

- MaxCut `p = 1`
- MaxCut `p = 2`

In addition, the current branch-kernel work means future optimisation work does
not need to restart from the raw Basso equations. The next implementation stage
should begin from:

1. the exact statevector oracle in `src/qaoa.jl`,
2. the raw multilinear constraint oracle in `src/transfer_oracles.jl`,
3. the signed-order branch-kernel scaffold in `src/basso_finite_d.jl`, and
4. the WHT factorisation note in `.project/learning/15-wht-factorisation-discovery.md`.
- zero-angle baselines
- clause-sign conventions

That is the right place to be before attempting a faster recursion.

Meanwhile, the experimental MaxCut transfer port gives a separate optimisation
track: future work can now compare compact MaxCut transfer objects against the
exact finite-tree oracle without re-porting the upstream code each time.
