## 1. Scope & Objective

P1.1 builds the purely combinatorial foundation for QAOA on regular hypergraphs.
The QAOA expectation value for a single constraint depends only on the tree-like
neighbourhood of that constraint out to depth p — the **light-cone tree**. This
spec captures the tree's structure as counting functions, not as an explicit
in-memory tree.

## 2. Design Decisions

### 2.1 No Explicit Tree — Parameters + Counting Functions Only

The most significant design choice: **we do not materialise a tree data
structure**. There are no `Node` objects, no child pointers, no adjacency lists.

**Rationale**: The tree is fully characterised by the triple `(k, D, p)` because
every branch at the same depth is structurally identical (by regularity of the
hypergraph). The tensor contraction algorithm (P1.3) exploits this symmetry — it
works with tensors indexed by depth level, contracting one representative branch
and raising the result to the branching multiplicity. An explicit tree would be
wasted memory and would obscure this symmetry.

The `TreeParams` struct plus its counting functions are the *complete* tree
interface. Downstream consumers (P1.2, P1.3) never need more.

### 2.2 Struct with Inner Constructor Validation

```julia
struct TreeParams
    k::Int   # constraint arity (hyperedge size)
    D::Int   # variable degree (regularity)
    p::Int   # QAOA depth
end
```

The inner constructor enforces `k ≥ 2`, `D ≥ 2`, `p ≥ 1`. These are physical
constraints:
- `k = 1` would be a trivial single-variable constraint (no XORSAT structure)
- `D = 1` would mean each variable appears in only one constraint (tree-like
  graph with no light-cone expansion)
- `p = 0` would be a zero-depth circuit (no QAOA rounds)

Using an **immutable struct** (Julia default) is deliberate — tree parameters are
fixed once created and should never be mutated during a computation.

### 2.3 0-Indexed Levels, Matching the Spec

Level indices are 0-indexed to match the spec's mathematical notation:
- Variable levels: j = 0, 1, …, p-1
- Constraint levels: j = 0, 1, …, p-1

Where j = 0 is the root (1 constraint node at constraint level 0, k variable
nodes at variable level 0). This convention aligns directly with the formulas
in the spec tables, avoiding off-by-one translation errors.

Both `variable_count_at_level` and `constraint_count_at_level` validate their
level argument and throw `ArgumentError` for out-of-range values.

## 3. Mathematical Foundation

### 3.1 Tree Structure

The light-cone tree is a bipartite tree alternating between constraint and
variable nodes:

```
Level 0 (constraint): 1 root constraint node
    ↓ k children
Level 1 (variable):   k variable nodes
    ↓ (D-1) children each
Level 2 (constraint): k(D-1) constraint nodes
    ↓ (k-1) children each
Level 3 (variable):   k(D-1)(k-1) variable nodes
    ↓ (D-1) children each
Level 4 (constraint): k(D-1)²(k-1) constraint nodes
    ...
```

The asymmetry at the root (k children, not k-1) is because we're expanding
*from* the root constraint. At every subsequent constraint node, one of its k
edges leads back toward the root (the parent edge), leaving k-1 child edges.
Similarly, each non-root variable has D neighbours, one of which is its parent
constraint, leaving D-1 child constraints.

### 3.2 Branching Factor

The two-level branching factor (one constraint→variable step + one
variable→constraint step) is:

```
b = (D-1)(k-1)
```

This governs exponential growth. For our primary target (k=3, D=4): b = 6.
For MaxCut validation (k=2, D=3): b = 2.

### 3.3 Node Count Formulas

| Level | Type | Count |
|-------|------|-------|
| 0 (root) | Constraint | 1 |
| 2j, j ≥ 1 | Constraint | k(D-1) · b^(j-1) |
| 2j+1, j ≥ 0 | Variable | k · b^j |

Totals (summing over j):
- **Variable nodes**: `∑_{j=0}^{p-1} k · b^j = k · (b^p - 1)/(b - 1)` for b > 1
- **Constraint nodes**: `1 + ∑_{j=1}^{p-1} k(D-1) · b^(j-1)` for p ≥ 2
- **Leaf count**: `k · b^(p-1)` (variables at the deepest level)

The implementation uses the simple summation form (`sum(... for j in 0:p-1)`)
rather than closed-form geometric series. This is clearer and avoids edge cases
for b = 1. The performance cost is negligible — p is typically ≤ 10.

### 3.4 Relationship Between Levels and QAOA Rounds

The tree has `2p` physical levels (0 through 2p-1), but the counting functions
use a compressed indexing where j ranges over `0:p-1` for both variable and
constraint levels. The mapping is:

- Constraint level j → physical tree level 2j
- Variable level j → physical tree level 2j+1

This compressed indexing aligns with QAOA rounds: round ℓ (1-indexed) in P1.2/P1.3
corresponds to the ℓ-th constraint→variable pair counting from the root.

## 4. Implementation Details

### 4.1 Source Structure (89 lines)

The source file is intentionally minimal — seven functions plus one type:

```julia
TreeParams(k, D, p)                     # Struct + validating constructor
branching_factor(t)                     # → (D-1)(k-1)
variable_count_at_level(t, j)           # → k · b^j (with j=0 special case)
constraint_count_at_level(t, j)         # → 1 if j=0, else k(D-1) · b^(j-1)
total_variables(t)                      # → sum over variable levels
total_constraints(t)                    # → sum over constraint levels
total_nodes(t)                          # → variables + constraints
leaf_count(t)                           # → variable_count_at_level(t, p-1)
```

Every function is a one- or two-liner. The design philosophy is composability:
totals are defined in terms of per-level counts, and `leaf_count` reuses
`variable_count_at_level`. No logic is duplicated.

### 4.2 Docstrings

All exported functions carry full Julia docstrings documenting:
- Parameters and return types
- The mathematical formula
- Special cases (root level, b=1)

This enables `?TreeParams` help in the Julia REPL and generates documentation
automatically.

### 4.3 Module Wiring

`src/QaoaXorsat.jl` includes `tree.jl` and exports all public functions:

```julia
include("tree.jl")
export TreeParams
export branching_factor, variable_count_at_level, constraint_count_at_level
export total_variables, total_constraints, total_nodes, leaf_count
```

The remaining stubs for tensor network and QAOA evaluation remain as comments,
providing a roadmap for P1.2 and P1.3.

## 7. Design Rationale: What We Didn't Build

The spec explicitly calls out that we do **not** build an explicit tree:

> *"We never materialise the full tree in memory for the tensor contraction
> (which exploits symmetry). But the tree structure informs the contraction."*

This is a conscious architectural choice, not laziness. Some alternatives
considered:

| Alternative | Why Rejected |
|-------------|-------------|
| Explicit tree with `Node` objects | Wastes memory; obscures the symmetry that P1.3 exploits |
| Recursive tree builder | No consumer needs to traverse; counting functions suffice |
| Memoised count functions | Unnecessary — p ≤ 10 means sub-microsecond computation |
| Closed-form geometric series | Edge cases for b=1; summation is clearer and equally fast |

The entire P1.1 deliverable is 89 lines of source and 74 lines of tests. This
minimalism is a feature — it provides exactly the interface downstream specs
need, with no excess abstraction to maintain.

## 8. How P1.2 and P1.3 Use This

### P1.2 (Tensor Network)

P1.2 uses `TreeParams` only indirectly — it needs `k` and `p` to size tensors:
- `4^p` = hyperindex dimension (from `p`)
- `k` = number of legs on problem/observable tensors
- The `D` parameter doesn't appear in individual tensors, only in contraction
  multiplicities

### P1.3 (Contraction)

P1.3 will use the full API:
- `branching_factor(t)` and `(t.D-1)`, `(t.k-1)` for exponentiation multiplicities
- `leaf_count(t)` to know how many boundary terms to initialise
- The level structure to sequence contraction from leaves (round p) to root (round 1)

The counting functions also serve as sanity checks — P1.3 can verify that
intermediate tensor counts match expected node counts at each level.
