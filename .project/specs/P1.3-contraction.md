**Depends on**: P1.1 (Factor Graph Tree), P1.2 (Tensor Network)

## Goal

Implement a correctness-first evaluator for a single-constraint QAOA
expectation value, and document the exact blocker that remains before the
intended `O(4^p)` branch-transfer contraction can be implemented safely.

## Background

The key insight from Farhi et al. 2025: on a regular tree, all branches at the
same depth are identical. Instead of contracting the full tree (which would require
exponential memory in the tree size), we contract a **single representative
branch** and use **element-wise exponentiation** to account for the multiplicity
of identical branches.

### The algorithm for MaxCut (k=2)

For a D-regular graph at depth p:

1. Start with the **leaf tensor** L (a vector of length 4^p)
2. **Contract inward** one level: combine L with the mixer/problem tensors for
   round p, yielding a branch tensor B_p
3. **Exponentiate**: $B_p \leftarrow B_p^{D-1}$ (element-wise) — accounts for
   (D-1) identical branches meeting at each variable node
4. **Contract one more level inward**: combine B_p with the next layer's
   problem/mixer tensors, yielding B_{p-1}
5. **Repeat** exponentiation and contraction until reaching the root edge
6. **Apply the observable tensor** at the root to get the scalar expectation value

Cost: O(p · 4^p) time, O(4^p) space.

### Extension to k-XORSAT (k ≥ 3)

For k-XORSAT on D-regular k-uniform hypergraphs, the tree is bipartite:
- Variable nodes (degree D — one parent constraint + D-1 child constraints)
- Constraint nodes (degree k — one parent variable + k-1 child variables)

The contraction alternates between two types of steps:

**Variable-to-constraint contraction** (going from leaves toward root):
1. Take a branch tensor B representing a subtree rooted at a variable node
2. This variable connects to (D-1) child constraint nodes, each with identical subtrees
3. Exponentiate: $B \leftarrow B^{D-1}$ (accounting for the D-1 identical branches)
4. Apply the mixer tensor for the appropriate round

**Constraint-to-variable contraction**:
1. Take the child branch message for a variable subtree
2. Contract a multilinear `(k-1)`-leg constraint update against the raw problem tensor
3. Use symmetry only to reuse the same child message on each child leg

Important: the draft rule `B \leftarrow B^{k-1}` at a non-root constraint is
not correct in general. Only the variable-node multiplicity exponentiates
entrywise.

The contraction proceeds from leaves to root, alternating between these two steps.

### Cost analysis

Each step processes a tensor of 4^p entries. There are 2p steps (p variable steps
+ p constraint steps). The exponentiation at each step is element-wise (O(4^p)).

The exact `O(4^p)` transfer object that achieves this cost is still open in this
repository. Until it is derived, the implementation uses a guarded exact
light-cone reference evaluator for small trees.

## Design

### Reference implementation pipeline

```julia
"""
Evaluate the QAOA expectation value for a single constraint.

Returns the expected fraction of the constraint being satisfied,
given tree parameters (k, D, p) and QAOA angles (γ, β).

This is the main entry point for the core computation.
"""
function qaoa_expectation(params::TreeParams, angles::QAOAAngles)::Float64
    parity = parity_expectation(params, angles)
    return 0.5 * (1 + parity)
end
```

### Exact light-cone reference steps

```julia
1. Build the explicit finite light cone of the root clause.
2. Prepare the uniform `|+⟩^{\otimes n}` state on its qubits.
3. Apply the physical QAOA rounds in forward order.
4. Measure the root-clause parity or satisfaction observable.
5. Refuse trees above the exact-reference guard threshold.
```

### Ordering requirement that remains valid

Any future transfer recursion must preserve the now-established ordering:

- physical round `1` is outermost
- physical round `p` is innermost
- root-to-leaf slice index satisfies `slice = p - round + 1`

## Implementation Notes

### Memory

The branch tensor has 4^p entries of Float64 = 8 × 4^p bytes.

| p | 4^p | Memory |
|---|-----|--------|
| 5 | 1,024 | 8 KB |
| 10 | 1,048,576 | 8 MB |
| 12 | 16,777,216 | 128 MB |
| 15 | 1,073,741,824 | 8 GB |

At p=12, this is very comfortable. At p=15, it needs a workstation with 16+ GB RAM.

### Numerical precision

- Element-wise exponentiation can cause numerical issues:
  - Entries near zero raised to large powers → underflow
  - Entries > 1 raised to large powers → overflow
- For (k=3, D=4): exponents are D-1=3 and k-1=2, which are moderate
- Use `log`-space computation if overflow/underflow becomes an issue:
  `b^n = exp(n * log(b))` is more numerically stable for large n

### Performance

- The inner loop is `branch .^ exponent` and matrix-vector products
- Julia's broadcasting (`.^`) is efficient for this
- At large p, consider using `@inbounds` and `@simd` annotations
- The computation is embarrassingly parallel over different angle evaluations
  (for optimisation), but each single evaluation is sequential

### Future transfer recursion

The P1.2 implementation provides `mixer_tensor(β, round, p)` as a full
`4^p × 4^p` matrix. A future optimised recursion must **not** materialise this
inside its inner loop. At p=12 that matrix would be 16M × 16M.

Instead, any future `apply_mixer!` should operate **in-place** on the branch
vector by iterating over all hyperindex values and mixing only the 2 bits for
the requested slice.

Cost: O(4^p) — same as the branch size. No matrix allocated.

Similarly, any future `apply_problem_gate!` should multiply each entry by the
appropriate phase based on the parity of the requested slice's bits.

The existing `mixer_tensor` and `problem_tensor` remain useful as **test
oracles** for such a future implementation.

## Correctness Ordering

The ordering issue is no longer open in the branch implementation. The exact
reference evaluator and `.project/learning/05-tensor-derivation.md` fix the
round-to-slice convention used throughout the code.

### What we know

1. The QAOA circuit applies rounds 1, 2, ..., p sequentially (round 1 first).
2. The tree has p variable levels and p constraint levels.
3. The contraction processes the tree from leaves (deepest level) to root.
4. Round ℓ's problem gate corresponds to the constraint node at some depth.
5. Round ℓ's mixer gate corresponds to the variable node at some depth.

The remaining open question is narrower: how to convert the raw P1.2 tensors into
the effective transfer object needed for the `O(4^p)` branch recursion without
double counting boundary or mixer contributions.

## Test Plan

### Validation tests (the critical ones)

```julia
@testset "MaxCut validation" begin
    # k=2, D=3, p=1: known optimal ≈ 0.69245
    params = TreeParams(2, 3, 1)

    γopt = atan(1 / sqrt(2))
    βopt = π / 8
    angles = QAOAAngles([γopt], [βopt])
    @test qaoa_expectation(params, angles; clause_sign = -1) ≈
        0.5 + sqrt(3) / 9 atol=1e-10
end
```

### Property tests

```julia
@testset "expectation_properties" begin
    params = TreeParams(3, 4, 1)

    @testset "bounded" begin
        for _ in 1:100
            γ = rand() * 2π
            β = rand() * 2π
            angles = QAOAAngles([γ], [β])
            result = qaoa_expectation(params, angles)
            @test 0.0 ≤ result ≤ 1.0
        end
    end

    @testset "zero_angles" begin
        # At γ=0: no problem phases → uniform superposition → random guess
        angles = QAOAAngles([0.0], [0.0])
        result = qaoa_expectation(params, angles)
        @test result ≈ 0.5 atol=1e-10  # random assignment: 50% satisfied
    end

    @testset "symmetry" begin
        # Periodicity: γ → γ + 2π should give same result
        γ, β = 0.3, 0.7
        r1 = qaoa_expectation(params, QAOAAngles([γ], [β]))
        r2 = qaoa_expectation(params, QAOAAngles([γ + 2π], [β]))
        @test r1 ≈ r2 atol=1e-10
    end
end
```

### Performance benchmarks (not tests, but instrumented)

```julia
# Track time and memory for increasing p
for p in 1:12
    params = TreeParams(3, 4, p)
    angles = QAOAAngles(rand(p), rand(p))
    @time qaoa_expectation(params, angles)
end
```

## Acceptance Criteria

1. `qaoa_expectation(TreeParams(2, 3, 1), optimal_angles)` ≈ 0.7500
2. Expectation values are always in [0, 1] for any angles
3. Zero angles yield 0.5 (random-guess baseline)
4. Results are insensitive to numerical noise (stable to at least 6 decimal places)
5. Oversized exact calculations fail with a clear guard message instead of an
    uncontrolled statevector blow-up
6. Contraction ordering is documented in `.project/learning/05-tensor-derivation.md`
7. The spec no longer claims the incorrect `.^ (k-1)` constraint update rule
