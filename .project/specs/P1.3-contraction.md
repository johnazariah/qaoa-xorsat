## Goal

Implement the core O(4^p) contraction algorithm that evaluates the QAOA
expectation value for a single constraint, using the element-wise exponentiation
trick on the symmetric tree. This is the computational heart of the project.

## Background

The key insight from Farhi et al. 2025: on a regular tree, all branches at the
same depth are identical. Instead of contracting the full tree (which would require
exponential memory in the tree size), we contract a **single representative
branch** and use **element-wise exponentiation** to account for the multiplicity
of identical branches.

### The algorithm for MaxCut (k=2)

For a D-regular graph at depth p:

1. Start with the **leaf tensor** L (a vector of length 4^p)
2. **Contract inward** one level: combine L with the mixer/problem tensors for
   round p, yielding a branch tensor B_p
3. **Exponentiate**: $B_p \leftarrow B_p^{D-1}$ (element-wise) — accounts for
   (D-1) identical branches meeting at each variable node
4. **Contract one more level inward**: combine B_p with the next layer's
   problem/mixer tensors, yielding B_{p-1}
5. **Repeat** exponentiation and contraction until reaching the root edge
6. **Apply the observable tensor** at the root to get the scalar expectation value

Cost: O(p · 4^p) time, O(4^p) space.

### Extension to k-XORSAT (k ≥ 3)

For k-XORSAT on D-regular k-uniform hypergraphs, the tree is bipartite:
- Variable nodes (degree D — one parent constraint + D-1 child constraints)
- Constraint nodes (degree k — one parent variable + k-1 child variables)

The contraction alternates between two types of steps:

**Variable-to-constraint contraction** (going from leaves toward root):
1. Take a branch tensor B representing a subtree rooted at a variable node
2. This variable connects to (D-1) child constraint nodes, each with identical subtrees
3. Exponentiate: $B \leftarrow B^{D-1}$ (accounting for the D-1 identical branches)
4. Apply the mixer tensor for the appropriate round

**Constraint-to-variable contraction**:
1. Take a branch tensor B representing a subtree rooted at a constraint node
2. This constraint connects to (k-1) child variable nodes, each with identical subtrees
3. Exponentiate: $B \leftarrow B^{k-1}$ (accounting for the k-1 identical branches)
4. Apply the problem gate tensor for the appropriate round

The contraction proceeds from leaves to root, alternating between these two steps.

### Cost analysis

Each step processes a tensor of 4^p entries. There are 2p steps (p variable steps
+ p constraint steps). The exponentiation at each step is element-wise (O(4^p)).

**Total cost: O(p · 4^p) time, O(4^p) space** — the same as MaxCut, independent
of k and D.

This is the crucial result: k and D only affect the exponents in the element-wise
exponentiation steps, not the tensor dimensions.

## Design

### The contraction pipeline

```julia
"""
Evaluate the QAOA expectation value for a single constraint.

Returns the expected fraction of the constraint being satisfied,
given tree parameters (k, D, p) and QAOA angles (γ, β).

This is the main entry point for the core computation.
"""
function qaoa_expectation(params::TreeParams, angles::QAOAAngles)::Float64
    depth(angles) == params.p ||
        throw(ArgumentError("angle depth must match tree depth"))

    # 1. Start from leaf tensor
    branch = leaf_tensor(angles)

    # 2. Contract from leaves to root, alternating var↔constraint steps
    for round in params.p:-1:1
        # Variable step: exponentiate for (D-1) identical subtrees,
        # then apply mixer
        branch = contract_variable_step(branch, angles.β[round], params)

        # Constraint step: exponentiate for (k-1) identical subtrees,
        # then apply problem gate
        branch = contract_constraint_step(branch, angles.γ[round], params)
    end

    # 3. Apply the root observable
    return apply_root_observable(branch, params)
end
```

### Individual contraction steps

```julia
"""
Contract one variable-node level of the tree.

Takes a branch tensor (representing the subtree below a variable node),
exponentiates it element-wise by (D-1) to account for identical branches,
then applies the mixer tensor for this round.
"""
function contract_variable_step(
    branch::Vector{Float64},
    β::Float64,
    params::TreeParams
)::Vector{Float64}
    # Element-wise exponentiation: account for (D-1) identical child branches
    branch = branch .^ (params.D - 1)

    # Apply mixer tensor for this qubit at this round
    return apply_mixer(branch, β, params.p)
end

"""
Contract one constraint-node level of the tree.

Takes a branch tensor (representing the subtree below a constraint node),
exponentiates it element-wise by (k-1) to account for identical branches,
then applies the problem gate tensor for this round.
"""
function contract_constraint_step(
    branch::Vector{Float64},
    γ::Float64,
    params::TreeParams
)::Vector{Float64}
    # Element-wise exponentiation: account for (k-1) identical child branches
    branch = branch .^ (params.k - 1)

    # Apply problem gate tensor for this constraint at this round
    return apply_problem_gate(branch, γ, params.p)
end
```

## Implementation Notes

### Memory

The branch tensor has 4^p entries of Float64 = 8 × 4^p bytes.

| p | 4^p | Memory |
|---|-----|--------|
| 5 | 1,024 | 8 KB |
| 10 | 1,048,576 | 8 MB |
| 12 | 16,777,216 | 128 MB |
| 15 | 1,073,741,824 | 8 GB |

At p=12, this is very comfortable. At p=15, it needs a workstation with 16+ GB RAM.

### Numerical precision

- Element-wise exponentiation can cause numerical issues:
  - Entries near zero raised to large powers → underflow
  - Entries > 1 raised to large powers → overflow
- For (k=3, D=4): exponents are D-1=3 and k-1=2, which are moderate
- Use `log`-space computation if overflow/underflow becomes an issue:
  `b^n = exp(n * log(b))` is more numerically stable for large n

### Performance

- The inner loop is `branch .^ exponent` and matrix-vector products
- Julia's broadcasting (`.^`) is efficient for this
- At large p, consider using `@inbounds` and `@simd` annotations
- The computation is embarrassingly parallel over different angle evaluations
  (for optimisation), but each single evaluation is sequential

### CRITICAL: In-place mixer application

The P1.2 implementation provides `mixer_tensor(β, round, p)` as a full
`4^p × 4^p` matrix. **Do not use this in the contraction loop.** At p=12 that
matrix would be 16M × 16M = 2 petabytes.

Instead, implement `apply_mixer!(branch, β, round, p)` that operates **in-place**
on the branch vector by iterating over all hyperindex values and mixing only the
2 bits for the given round (ket_round, bra_round). Each hyperindex value σ
participates in a 4-element local operation (the 2×2 mixer on ket ⊗ conjugate
mixer on bra), leaving all other bits untouched.

Cost: O(4^p) — same as the branch size. No matrix allocated.

Similarly, `apply_problem_gate!(branch, γ, round, p)` should multiply each entry
by the appropriate phase based on the parity of the round's bits — also O(4^p)
and in-place.

The existing `mixer_tensor` and `problem_tensor` remain useful as **test
oracles**: verify that the in-place versions produce identical results at small p.

## Correctness Ordering — MUST RESOLVE BEFORE CODING

The contraction goes **leaves to root**. But the mapping from QAOA rounds to
tree levels — i.e. which round's γ and β apply at which contraction step — is
the single most critical design decision in the entire project. Getting it wrong
produces plausible but incorrect numbers.

### What we know

1. The QAOA circuit applies rounds 1, 2, ..., p sequentially (round 1 first).
2. The tree has p variable levels and p constraint levels.
3. The contraction processes the tree from leaves (deepest level) to root.
4. Round ℓ's problem gate corresponds to the constraint node at some depth.
5. Round ℓ's mixer gate corresponds to the variable node at some depth.

## Acceptance Criteria

1. `qaoa_expectation(TreeParams(2, 3, 1), optimal_angles)` ≈ 0.7500
2. Expectation values are always in [0, 1] for any angles
3. Zero angles yield 0.5 (random-guess baseline)
4. Results are insensitive to numerical noise (stable to at least 6 decimal places)
5. Performance scales as O(4^p) — doubling p should take ~4× longer, not more
6. Memory usage is O(4^p) — no hidden allocations scaling with tree size
7. Contraction ordering is documented in `.project/learning/05-tensor-derivation.md`
   with citations to Farhi 2025 equations
