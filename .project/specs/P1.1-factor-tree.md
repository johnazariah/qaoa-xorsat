## Goal

Implement the factor graph tree that represents the QAOA light cone for a single
constraint on a D-regular k-uniform hypergraph at depth p. This is a purely
combinatorial data structure — no quantum mechanics or tensors here.

## Background

The QAOA expectation value for a single constraint depends only on the tree-like
neighbourhood of that constraint out to depth p (assuming the hypergraph has girth
> 2p, which is the regime we work in). This neighbourhood is a bipartite tree
alternating between two node types:

- **Constraint nodes** (hyperedge factors) — each connects to k variable nodes
- **Variable nodes** — each connects to D constraint nodes (one parent + D-1 children)

At the root is the constraint we are evaluating. The tree has 2p levels (p pairs
of constraint→variable layers).

## Tree Structure for General (k, D, p)

```
Level 0: 1 root constraint node
         ↓ connects to k variable nodes
Level 1: k variable nodes
         ↓ each connects to (D-1) child constraint nodes
Level 2: k(D-1) constraint nodes
         ↓ each connects to (k-1) child variable nodes
Level 3: k(D-1)(k-1) variable nodes
         ↓ each connects to (D-1) child constraint nodes
Level 4: k(D-1)(k-1)(D-1) = k(D-1)²(k-1) constraint nodes
...
```

The branching factor per two-level step is `(D-1)(k-1)`.

### Node counts by level

Let `b = (D-1)(k-1)`.

The tree has 2p levels (0 through 2p-1):
- **Constraint levels**: 0, 2, 4, ..., 2(p-1) → p levels total
- **Variable levels**: 1, 3, 5, ..., 2p-1 → p levels total
- **Leaf nodes** are the deepest variable level (level 2p-1), in state |+⟩

**Exact formulas:**

| Level | Type | Count formula |
|-------|------|---------------|
| 0 | Constraint | 1 |
| 2j (j ≥ 1) | Constraint | k(D-1) · bʲ⁻¹ |
| 2j+1 (j ≥ 0) | Variable | k · bʲ |

**Totals:**
- Variable nodes: `k · (bᵖ - 1) / (b - 1)` for b > 1, or `k · p` for b = 1
- Constraint nodes: `1 + k(D-1) · (bᵖ⁻¹ - 1) / (b - 1)` for b > 1 and p ≥ 2, or `1` for p = 1
- Leaf count: `k · bᵖ⁻¹`

### Concrete sizes for (k=3, D=4)

Here b = (D-1)(k-1) = 3 × 2 = 6.

| p | Variable nodes | Constraint nodes | Total nodes |
|---|---------------|-----------------|-------------|
| 1 | 3 | 1 | 4 |
| 2 | 3 + 18 = 21 | 1 + 9 = 10 | 31 |
| 3 | 3 + 18 + 108 = 129 | 1 + 9 + 54 = 64 | 193 |
| 4 | 3 + 18 + 108 + 648 = 777 | 1 + 9 + 54 + 324 = 388 | 1165 |
| 5 | 4,665 | 2,332 | 6,997 |

### Golden values for MaxCut (k=2, D=3)

b = (3-1)(2-1) = 2.

| p | Variable nodes | Constraint nodes | Total | Leaf count |
|---|---------------|-----------------|-------|------------|
| 1 | 2 | 1 | 3 | 2 |
| 2 | 2 + 4 = 6 | 1 + 4 = 5 | 11 | 4 |
| 3 | 6 + 8 = 14 | 5 + 8 = 13 | 27 | 8 |
| 4 | 14 + 16 = 30 | 13 + 16 = 29 | 59 | 16 |

These two tables are the **authoritative test data**. Every value must match.

Note: we never materialise the full tree in memory for the tensor
contraction (which exploits symmetry). But the tree structure informs the contraction.

## Design

### Types

```julia
"""
Parameters defining a regular hypergraph QAOA light-cone tree.

- `k`: constraint arity (hyperedge size). k=2 is MaxCut.
- `D`: variable degree (regularity).
- `p`: QAOA depth (number of rounds).
"""
struct TreeParams
    k::Int
    D::Int
    p::Int

    function TreeParams(k::Int, D::Int, p::Int)
        k ≥ 2 || throw(ArgumentError("k must be ≥ 2, got $k"))
        D ≥ 2 || throw(ArgumentError("D must be ≥ 2, got $D"))
        p ≥ 1 || throw(ArgumentError("p must be ≥ 1, got $p"))
        new(k, D, p)
    end
end
```

### Key functions

```julia
"""Branching factor per two-level step: (D-1)(k-1)."""
branching_factor(t::TreeParams) = (t.D - 1) * (t.k - 1)

"""Number of variable nodes at a given variable level j (0-indexed, j=0..p-1)."""
variable_count_at_level(t::TreeParams, j::Int) =
    j == 0 ? t.k : t.k * branching_factor(t)^j

"""Number of constraint nodes at a given constraint level j (0-indexed, j=0 is root)."""
constraint_count_at_level(t::TreeParams, j::Int) =
    j == 0 ? 1 : t.k * (t.D - 1) * branching_factor(t)^(j - 1)

"""Total number of variable (qubit) nodes in the tree."""
total_variables(t::TreeParams) =
    sum(variable_count_at_level(t, j) for j in 0:t.p-1)

"""Total number of constraint nodes in the tree."""
total_constraints(t::TreeParams) =
    sum(constraint_count_at_level(t, j) for j in 0:t.p-1)

"""Total nodes (variables + constraints)."""
total_nodes(t::TreeParams) = total_variables(t) + total_constraints(t)

"""Number of leaf nodes (variable nodes at the deepest level)."""
leaf_count(t::TreeParams) = variable_count_at_level(t, t.p - 1)
```

### What we do NOT build

We do **not** build an explicit tree data structure with node objects and pointers.
The tree is fully characterised by `(k, D, p)` — every branch at the same depth
is identical (by regularity). The contraction algorithm (Spec P1.3) works with
tensors indexed by depth level, not with explicit tree nodes.

The `TreeParams` type and its functions are the complete interface for tree queries.

## Test Plan

### Unit tests

```julia
@testset "TreeParams" begin
    @testset "construction" begin
        @test TreeParams(2, 3, 1) isa TreeParams
        @test TreeParams(3, 4, 5) isa TreeParams
        @test_throws ArgumentError TreeParams(1, 3, 1)  # k < 2
        @test_throws ArgumentError TreeParams(2, 1, 1)  # D < 2
        @test_throws ArgumentError TreeParams(2, 3, 0)  # p < 1
    end

    @testset "branching_factor" begin
        @test branching_factor(TreeParams(2, 3, 1)) == 2   # MaxCut D=3
        @test branching_factor(TreeParams(3, 4, 1)) == 6   # Our target
        @test branching_factor(TreeParams(4, 5, 1)) == 12
    end

    @testset "node_counts k=2, D=3 (MaxCut)" begin
        t = TreeParams(2, 3, 1)
        @test total_variables(t) == 2      # just the 2 endpoints
        @test total_constraints(t) == 1    # just the root edge
        @test total_nodes(t) == 3
    end

    @testset "node_counts k=3, D=4" begin
        t1 = TreeParams(3, 4, 1)
        @test total_variables(t1) == 3
        @test total_constraints(t1) == 1
        @test total_nodes(t1) == 4

        t2 = TreeParams(3, 4, 2)
        @test total_variables(t2) == 21    # 3 + 18
        @test total_constraints(t2) == 10  # 1 + 9

        t3 = TreeParams(3, 4, 3)
        @test total_variables(t3) == 129
    end

    @testset "leaf_count" begin
        @test leaf_count(TreeParams(2, 3, 1)) == 2
        @test leaf_count(TreeParams(3, 4, 1)) == 3
        @test leaf_count(TreeParams(3, 4, 2)) == 18
    end

    @testset "monotonicity" begin
        for k in [2, 3, 4], D in [3, 4, 5]
            for p in 1:4
                tp = TreeParams(k, D, p)
                tp_next = TreeParams(k, D, p + 1)
                @test total_variables(tp_next) > total_variables(tp)
                @test total_nodes(tp_next) > total_nodes(tp)
            end
        end
    end
end
```

## Acceptance Criteria

1. `TreeParams(k, D, p)` validates inputs and stores parameters
2. All count functions produce correct values for hand-verified cases
3. Node counts match the table above for (k=3, D=4) at p=1..5
4. MaxCut (k=2, D=3) produces correct counts
5. All tests pass
